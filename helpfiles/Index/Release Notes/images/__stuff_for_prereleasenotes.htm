Overview

The ink control, shown in the image below, is made up of two main parts, the editor and the view.

 

 The editor (bottom part) allows the user to edit the ink while the view (top part) provides the ability to navigate the ink. The view contains a drag-able zoom-box (see the dotted red rectangle) showing the portion of the ink visible in the editor. The editor and view can be laid out in three ways.

The first is a split layout. In this layout the editor and the view share the ink controls layout area in either a horizontal or vertical layout. It should be noted that a split layout can still optionally show only the view or editor. This is useful if you are targeting multiple screen sizes, as you can use responsive layout Javascript to change the split layout on smaller screens to toggle between the editor and view if there isn't space for both. You can also use this to allow the user to toggle the editor to take up the entire ink control on larger screens as well.

The second layout is a detached view layout. In this layout the editor takes up the entire ink control area, while the view part is injected into an element specified by an ID. This allows for extremely customized layouts.

The last layout is an editor layout. This will display only the editor. This is useful for ink editing that doesn't need navigation, such as signature capture.

Ink data is stored in a string format, with the strokes stored as vectors. Below is an example of some ink data.

This keeps the file size down and allows for the ink to be rendered at any resolution. The boundaries of the ink page are defined by the ink page settings. The page settings include width and height in a specified unit (in, px, etc.). This data is combined with the dots-pre-unit (dpu) to create the ink page. The width and height are multiplied by the dpu to get the base page size in "ink units". The dpu can be thought of like "dpi" in printing. The higher the dpu, the higher the ink resolution.

The goal is to get the dpu relative to the units so that when the editor is at a scale of '1 ink unit equals 1px', the user is encouraged to enter ink at a larger scale than if they where writing on a sheet of paper. This works around the inherent lack of accuracy in capacitive touch screens (ink can only be captured at screen pixel accuracy), and allows for readable written input.

If the ink page unit is 'inches' then the dpu should be at least be 300. All measurements inside of the ink (pen width, margins, etc.) are in 'ink units' derived from the dpu.

The page can also store background (color and image) settings, and have automatically generated lines to create graph and ruled paper look.

The editor and view parts of the ink control can each have an overlay. The overlay is arbitrary HTML that will be laid out over each part. Elements in the HTML can make use of the "a5-item" attribute in order to easily add buttons to the ink control. There are a number of predefined "items" that allow the user to access the eraser tool, pan tool, undo and redo actions, and navigate the ink. Custom "items" allow you to add unique behaviors to elements in the overlay. The events on the "items" have direct access to the ink control.

The ink editor uses tools to edit the ink. The default tool in the pen tool. Other tools can be selected using the ink control's .setTool() method. This method can specify whether the tool is "sticky" or not. If a tool is not "sticky" then after a single user interaction with the tool, the tool will be deselected, and the ink editor will go back to the pen tool. Each tool has an optional "active class name" that will be applied to the ink control when that tool is active. This allows you, though CSS, to automatically highlight the elements in the overlay, and possibly change visibility of elements as well.

UX Component - Multi-Lingual Applications - Client-Side Translation - In a typical Alpha Anywhere web application, multi-language applications are constructed by wrapping language tags ( <a5:r> ) or text dictionary tags ( <a5:t> ) around the strings that you want to translate. Then, when the component is served by the server, the strings are translated before the HTML or Javascript is sent to the client.

However, in a PhoneGap application, the application is loaded from a file that is installed on the device and there is no callback to the server to load the component (where language translation would have occurred). If you would like to allow the user to set a preferred language in a PhoneGap application you need to be able to change the strings in the application using client-side Javascript.

This topic describes how this can be done.

First you will need to wrap the strings in your application that need to be dynamically translated in <span> tags and you will need to give each of these spans a unique id. For example, assume that your application has a label that displays 'Firstname' and another label that displays 'Lastname'. You would change the label text for these two labels to something like:

<span id="_string_firstname">Firstname</span>

<span id="_string_lastname">Lastname</span>

 

Now that each Label has a unique Id, it is easy to write Javascript to change the Label. For example, the Javascript to change the Firstname label to (say) 'Enter your first name' would be

$('_string_firstname').innerHTML = 'Enter your first name';

 

So, when the user selects a different preferred language, you would need to loop through all of the ids that need to be updated and set the innerHTML to the text in the preferred language.

In order to get the text in the selected language you would need to define a Javascript object that has all of the strings that need to be translated and the value of that string in each of the possible languages that the user could select. If you were working with multi-byte languages, you will also need the unicode encoded value of the srtring.

This Javascript object could either be defined in a Javascript file (e.g. a .js file) that is linked to the component (you would specify this in the Javascript Linked files property in the UX builder), or it could be defined in the component itself (for example in the onRenderComplete client-side event).

when to use a viewbox and when to use a list

 

if you have complex data (e.g. nested json documents) -- i.e. not just a array of itenentical objects --- use a view box

list that is simple and don't need things like filter, order, groupd, group navigator, virtualization, scroller, columnar view

 

use a view box to do complex hit targets --- view box can be populated with arbitrary html OR svg and you can associate events with elements in the html or svg without have to bind events to EVERY element and you don't need to unbind events when you swap out the html. you can also make any element 'selectable' -- much easier than manually add/remove classes from emelement. -- can also navigate through the selectable items - would be very hard to do if you manually made items selectable with your own code. also support multi-select and the concept of a 'value'

The next build will have support for a new keyword in portable SQL AFTER.

 

For example:

 

            SELECT FIRST 100 AFTER 1000 * from Customers order by CustomerId

 

The reason I say experimental is that:

1.       Not all databases support offsets

2.       Ive lightly tested SQL Server and MySQL as well as Access (which does NOT support it), but there are a lot of permutations still to be tested and Ive already gone beyond my mandate.

 

Details

 

1.       The portable SQL parser now supports the AFTER keyword/clause as in FIRST n AFTER m

2.       There is a new function on the SQL::Connection object to test for support of the AFTER keyword for a specific syntax and database version (both arguments are defaulted to the current connection values.

?cn.SelectAfterSupported()

?cn.SelectAfterSupported(Oracle)

?cn.SelectAfterSupported(SQLServer, "2012")

 

3.       Note that SQL Server only supports the OFFSET syntax for SQL Server 2012 and 2014, so prior versions will not support AFTER.  When the native syntax is generated, if the SQL does NOT have an order-by clause, the older syntax (TOP n) is generated.

4.       When generate SQL, if a database does not support the AFTER/offset syntax, the generated SQL will ignore the value and generate SQL as if it was not there.  This behavior can be changed, but it seemed the least likely approach to break existing applications.

 

Database syntaxes that will attempt to support AFTER include:

 

A5SQL Portable SQL

EnterpriseDB

Generic

MySql

Oracle

OracleCaseSensitive

PostgreSQL

PostgresPlus

SQLAnywhere

SQLAzure

SQLite

SQLServer (2012, 2014 and only if there is an order-by clause)

 

 

 

Sample Script

sql = "SELECT first 12 * FROM customers order by customerID"

sql = "SELECT first 12 AFTER 5 * FROM customers order by customerID"

sql = "SELECT first 12 AFTER 5 * FROM customers"

 

 

dim cn as sql::connection

?cn.open("::Name::SQLServer2014Northwind")

?cn.execute(sql)

 

x = cn.resultset.tostring()

 

?cn.SelectAfterSupported()

A5Storage::SubContainer::CopyItemTo()

A5Storage::DataItem::CopyTo()

function odraw_remap as c(line as c)
line = strtran(line,"].","],")
dim offset as n = 0
dim levels as n = occurs(",",line)
if levels > 0 then
offset = 0.09 * levels
end if
if right(line,1) = "]" then 
line = "{data="+line+"}{T="+offset+"}  {I:'$$control.box'}"+word(line,-1,",")
else
line = "{data="+line+"}{T="+offset+"}  {I:'$$control.text'}"+word(line,-1,",")
end if
odraw_remap = line
end function
