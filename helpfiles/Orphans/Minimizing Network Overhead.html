Minimizing Network Overhead

Overhead considerations when designing for a networked application

Remember records are ?processed' on the local machine, which means data has to pass through the network to the local machine. Ranges and queries built upon Lightning Query Optimization only need to pass the index, not the records, over the network. Think of an index and/or query as a one-to-one relationship with the table.

Open tables and files in read only mode whenever possible.

Remember, every form opened up has all its underlying tables opened also.

Try not to use referential integrity. If you need referential integrity, create a special set for use only when it is absolutely necessary.

When designing indexes:

  • Do not use filters. Use a range instead. (Simple filters should not cause problems.)

  • Use simple field values for your index key values.

  • Do not use calculated fields or expressions that are subject to frequent changes. Make sure the key value of an index expression is not subject to change unless the record is open in enter or change mode.

  • Never use variable values as all or part of an index expression.

One to many sets create multiple times more overhead than single tables and one to one sets.

  • This subjective recommendation must take into account how many tables, how many records, how many indexes, how many linking values are constructed, referential integrity, etc.

  • Keep sets simple.

  • Use multiple specific simple sets for your needs, rather than a large ?all encompassing' set.

Linking keys should be based on simple field values rather than complex expressions or calculated fields.

Always if possible, use a simple "meaningless" field (e.g. an auto incremented field) value as the linking key.

Avoid calculated fields that reference tables or indexes.

Use queries that are local (saved in a private directory) as opposed to indexes which are global (maintained on a server).

The more enter/edit sessions open, the more overhead.

The more nodes on a network doing data entry, the more overhead.

Techniques to use if necessary

Only open as many tables and sets as you need.

Use forms based on a child table only to add records to a child table of a set, while opening the set in read only mode.

When entering a new record, programmatically begin a new record and save it immediately, then present it to the user in change mode.

For some reporting and data intensive routines, consider creating temporary files locally by doing a copy command. Then processed records can either be disposed of or written in a batch to the server files.

Keep tables open in enter mode for the shortest time possible.

Key values only needed for data intensive critical routines might be created, built, used, and disposed of in the Xbasic script.

Programmatically monitor, create, and or rebuild your indexes.

Keep memo fields to a minimum and located in a separate linked table if possible. Putting the memo field in a separate table also allows you the ability to open this table separately, save an empty memo, then open it up in change mode for the data entry. The reason for this is that the indexes are written almost immediately so allows little opportunity for index and network problems.

In all things be aware of the network ?traffic' that your routine will cause and try to keep that to a minimum.
