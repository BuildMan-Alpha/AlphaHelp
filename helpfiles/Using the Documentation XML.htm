Using the Documentation XML

Introduction:

When creating new pages for Alpha Anywhere's Help Documentation it is vastly preferable to use Alpha's XML documentation notation, as opposed to HTML. Being strictly metadata, the XML format focuses solely on content, thereby eliminating the need to consider the styling of a given page. While at first this might seem to be a limitation of artistic expression it has a great many advantages.

Styles change quickly and can often date a page to the year or decade, in which it was created. As years pass it eventually becomes necessary to update the page styling. When dealing with an individual document this is not a problem. However, when dealing with thousands of separate documents it becomes preferable to have one styling template that governs all pages from one location.  This makes it possible to change the look and feel of the entire documentation from one location, rather than muck about with hundreds of CSS pages buried in individual folders or ,even worse, styling that is defined in the html itself .

Using the XML notation ensures that people who are doing documentation follow a format that is roughly similar format across documents, not just in style but also structure. There is flexibility in using the XML but it does operate using a limited set of rules. This makes the documentation as a whole much easier to maneuver for both users and programmers. It also allows for the manipulation of the metadata contained in the XML to create new features in the help system, without having to alter the existing XML page.

The XML format also has provisions for including code from other pages, adding custom shortlinks that avoid name collisions for permalinks, dynamic URL lookups for "see also" references, and more.

Creating a new XML page

Use <page></page> tags to create a new page. All of the page's content will go between these tags.

 

Add a Page Title or Topic

Use the <topic></topic> tags to create a document heading

Example


<page>
    <topic>Using the Documentation XML</topic>
</page>

Adding a Shortlink

Shortlinks are used to create permalinks for pages in documentation. The shortlink is a link that is used to dynamically look up a page. It is the prefered method for sharing pages, as pages in documentation can be moved. The URL in the browser shows the literal location of the document on the server. If the file is moved, this URL will change and anyone who has a copy of the original link will no longer be taken to the correct page. A "share" feature was added to alleviate this problem. By default, the shortlink for your page will use the page topic. However, in the event that a topic collides with another topic, the shortlink tag can be used to create a shortlin for your page.

Example


<page>
    <shortlink>Grid Component Code</shortlink>
    <topic>Code</topic>
    <description>Code section for the Grid Component</description>
</page>

Adding Sections to a page:

The <sections></sections> tags allow you to add multiple sections to a page. Only one pair of sections tags are needed in most documents, unless you are using <group> tags, but you are in no way limited to using two. You can use as many <sections> tags as you want and your document should still parse. These tags are useful only in that they contain the <section></section> tags.

Adding a Section:

The <section></section> tags fit inside the <sections></sections> tags and contain specified information or data.

Example:


<page>
    <topic>Using the Documentation XML</topic>
    <sections>
        <section>
        </section>
    </sections>
</page>

You can have as many <section></section> tag pairs in a <sections></sections> tag pair as you please.

Example:


<sections>
    <section>
    </section>
    <section>
    </section>
</sections>

Adding a Title to a Section:

You can add a title to a section by using the <title></title> tags.

Example:


<page>
    <topic>Using the Documentation XML</topic>
    <sections>
        <section>
            <title>Adding Content to a Section:</title>
        </section>
    </sections>
</page>

Adding Text to a Section:

You can add text to a section by using the <description></description> tags.

Example:


<page>
    <topic>Using the Documentation XML</topic>
    <sections>
        <section>
            <description>When you want to add text to a section do this</description>
        </section>
    </sections>
</page>

Adding Code Examples to a Section:

When you want to add code to a section, the <example></example> tag will help the code stand out from the text so that users can copy and paste it into Alpha Anywhere. Because the code often contains angle brackets <<>> and or an ampersand(s) & that will throw the xml parser, the <![CDATA[ ]]> tag is required. This tag reduces these characters to simple text. The <![CDATA[ ]]> tag can be used inside other tags as well.

Example:


<page>
    <topic>*ARRAY_APPEND Function</topic>
    <sections>
        <section>
        <example><![CDATA[dim b[1] as C
*array_append(b, "b1")
*array_append(b, "b2")
? b
= [1] = "b1"
[2] = "b2"
dim c[1] as N
*array_append(c, 123)
*array_append(c, 456)
? c
= [1] = 123
[2] = 456]]></example>
        </section>
    </sections>
</page>

NOTE: The example tag treats whitespace literally.

Add An Image to a Section:

To add an image to a section use the <figure><link></link></figure> tags and include the file path of the image in the middle.

The figure tag supports the following child tags:

<title></title>
Adds a caption below the image. Optional.
NOTE: If you add a title and leave it empty, this can result in all text after the figure to be centerd on the page. It is best to always remove empty xml tags from your document.
<alt></alt>
Alt text added to the HTML attributes for the figure. Eg, <img alt="this is where ALT text is placed">. Also optional.
<link></link>
The link to the figure. This is required.

Example:


<page>
    <topic>Google Map Component</topic>
    <sections>
        <section>
            <figure>
                <title>A map of the local area.</title>
                <alt>An image of a map</alt>
                <link>images/google_map_1.png</link>
            </figure>
        </section>
    </sections>
</page>

 

Add a Note to a Section:

A note is text that stands out from the rest of the document as either a warning or a tip to the user. Notes use the <note></note> tags and are displayed with a highlighted background once the XML code is parsed. They do not include images or code. For this either look to <groups> or <cases> tags. A note can be added to a <section></section> a <step></step> or a <case></case>.

You do not need to include the word 'Note:' in the beginning of your text as this is added automatically after parsing. Note tag pairs can contain paragraphs using the standard HTML <p></p> tags.

Example:


<page>
    <topic>Google Map Component</topic>
    <sections>
        <section>
            <note>You may see a warning about port 80. Click here to fix it.</note>
        </section>
    </sections>
</page>

Add a Warning to a Section:

A warning is useful in alerting readers about something. To create a warning use the <warning></warning> tag pair.

Example:


<page>
    <topic>Google Map Component</topic>
    <sections>
        <section>
            <warning>
                Do not use this function in a production
                environment with the reset property on!
            </warning>
        </section>
    </sections>
</page>

Add an Ordered List to a Section.

You can use the <steps></steps> tags to add an ordered list to a section. These tags operate much like the <sections></sections> tags.

Example:


<page>
    <topic>Ordered List</topic>
    <sections>
        <section>
            <steps></steps>
        </section>
    </sections>
</page>

Once you have a <steps></steps> tag pair you can add <step></step> tags for each numbered step in the ordered list. The numbers are added automatically when the xml is parsed.

Example:


<page>
    <topic>Ordered List</topic>
    <sections>
        <section>
            <steps>
                <step>
                </step>
            </steps>
        </section>
    </sections>
</page>

Add as many of these <step></step> tags required...


<steps>
    <step>
    </step>
    <step>
    </step>
    <step>
    </step>
</steps>

Each <step></step> pair can contain a title, description, example, image, list (see <list>), subsection (see <cases>), or note (see <note).

Example:


<page>
    <topic>Converting an Action to Inline Xbasic</topic>
    <sections>
        <section>
        <title></title>
            <steps>
                <step>
                    <description>Open the Lesson2 script in the Code Editor.</description>
                </step>
                <step>
                    <description>Select the second Action, "Display a Message Box".</description>
                </step>
                <step>
                    <description>Right-click and select Convert to Inline-Xbasic.</description>
                    <figure>
                        <link>images/converting_1.gif</link>
                    </figure>
                </step>
            </steps>
        </section>
    </sections>
</page>

Adding an Unordered List to a Section:

You can create an unordered list using the <list></list> tags. This creates a simple two column unordered list (possibly one column) containing <item></item> tags for each element in the list. Every item can contain a <name></name> tag pair (the first column) and a <description></description> tag pair (the second column). Add as many <item></item> tag pairs as needed.

Example:


<page>
<topic>Unordered List</topic>
    <sections>
        <section>
        <list>
            <item>
                <name>Quill Gordon</name>
                <description>Epeorus pluralis</description>
            </item>
            <item>
                <name>Hendrickson</name>
                <description>Ephemerella subvaria</description>
            </item>
        </list>
        </section>
    </sections>
</page>

Add -title to the end of  'name' or 'description' to create a distinct column heading.


<page>
    <topic>Unordered List</topic>
    <sections>
        <section>
            <list>
                <item>
                    <name-title>Common Name</name-title>
                    <description-title>Scientific Name</description-title>
                </item>
                <item>
                    <name>Quill Gordon</name>
                    <description>Epeorus pluralis</description>
                </item>
            </list>
        </section>
    </sections>
</page>

Add an Unordered Sub-Section to a Section:

You can add a sub-section to a <section></section> by using the <cases></cases> tag pair. The <cases></cases> tag pair can also be used inside <step></step> and even inside the <case></case> tag pairs; for occasions where you want to create subsections within subsections.

Examples:


<section>
    <cases>
    </cases>
</section>

Alternately....


<section>
    <steps>
    <step>
    <cases>
    </cases>
    </step>
    </steps>
</section>

Alternately....


<section>
    <steps>
        <step>
            <cases>
                <case>
                    <cases>
                    </cases>
                </case>
            </cases>
        </step>
    </steps>
</section>

As with <section> and <step>, the <case></case> tags contain the actual information embedded in the subsection. Every time you create a subsection, the content within it will appear to be slightly indented from any text inside the <section></section> tags that contain it. You can use the following tags inside <case></case>:

  • <title></title>
  • <description></description>
  • <cases></cases>
  • <steps></steps>
  • <figure><link></link></figure>
  • <note></note>
  • <example><![CDATA[ ]]><example>
  • <list></list>

Example:


<section>
    <steps>
        <step>
            <cases>
                <case>
                    <figure>
                        <link>images/converting_2.gif</link>
                    </figure>
                </case>
                <case>
                    <note>The action is called "Inline Xbasic" because this action type 
                    allows you to insert Xbasic within an Action Script.</note>
                </case>
            </cases>
        </step>
    </steps>
</section>

 

Adding Videos to a Section:

You can add <videos></videos> tags inside a <section></section> tag pair to create a container for video links. Each video link is then described using a <video></video> tag pair that contains <name> and </name> tags for the name of the video and <link></link> tags that contain the defined link to the video. Since this link is external you will need to add the file path in order for the link to work.

Example:


<page>
    <topic>Videos</topic>
    <sections>
        <section>
            <videos>
                <video>
                    <name>Watch Video</name>
                    <link>http://www.ajaxvideotutorials.com/V12Videos/json_xbasic_1.swf</link>
                </video>
            </videos>
        </section>
    </sections>
</page>

Adding Embedded Videos:

By adding embedded="true" to the inside of a video tag you can embed a video. This way you will see the image of the video and a play symbol in the document, just as you might see a youtube video.

Example:


<page>
    <topic>Videos</topic>
    <sections>
        <section>
             <videos>
                <video embedded="true"<>
                    <name>Watch Video</name>
                    <link>http://www.ajaxvideotutorials.com/V12Videos/json_xbasic_1.swf</link>
                </video>
            </videos> 
        </section>
    </sections>
</page>

 

Adding Groups to a Page

The groups tag, written <groups></groups>, operates in almost the same way that the <sections></sections> tag pair functions, except that <groups> is a level up from <sections>. In most instances the <groups> tag is not necessary. It is useful for writing tutorials when you want to have a background color block that contains useful background text. You can do this using the <group></group> tags described below.

Example:


<page>
    <topic>Groups</topic>
    <groups>
    </groups>
</page>

The <group></group> tag pair is contained within <groups></groups> and is where large scale content is stored. The only tags that <group> can contain are <title></title> and <sections></sections>. The group title is a level above a section title and at some point might be used to generate a structured table of contents automatically from any given xml page.

Example:


<page>
    <topic>Groups</topic>
    <groups>
        <group>
        </group>
    </groups>
</page>

To add color to a group simply add background="true" to a group tag, everything inside the group will have a background color.

Example:


<page>
    <topic>Groups</topic>
    <groups>
        <group background="true">
        </group>
    </groups>
</page>

Automatically Titled Sections

Adding a See Also Reference List

The <see></see> tags create a See Also section in your document that can contain references that link to related documents. These tags should be placed at the end of your document, just before the </page> tag.

Example:


<page>
    <topic>Create a See Also</topic>
    <see>
    </see>
</page>

Add a Reference to a Reference List:

The <ref></ref> tags can be used to add an single reference that links to another document. You do not need to worry about adding actual file path, just the name of the document that you want to link to. The help documentation will handle the rest. You can add as many references as needed.

Example:


<page>
    <topic>Create a See Also</topic>
    <see>
        <ref>Displaying the Main Menu</ref>
        <ref>Displaying the Navigation Toolbar</ref>
    </see>
</page>

The Limitations Section:

The <limitations></limitations> tag pair adds a Limitations section to your document that will automatically be titled 'Limitations'. This section frequently contains information such as Desktop Applications Only or Web Applications Only. The <limitations></limitations> tags go at the end of a document just before the <see> tag.

Example:


<page>
    <topic>Adding a Limitations Tag</topic>
    <limitations>Desktop Applications Only<limitations>
    <see>
        <ref>Displaying the Main Menu</ref>
    </see>
</page>

Here are several examples of limitations used in documentation. This is not a definitive list of limitations.

  • Desktop Applications Only
  • Web Applications Only
  • Web and Mobile Applications Only
  • iOS Applications Only
  • PhoneGap Build Applications Only
  • Layout Table Reports Only

The Description Section:

As noted earlier, you can use the <description></description> tags to add text to a specified section in a document. You can also use this tag by itself to add a designated description or summary to a document. This description then describes the information the document contains. This information will also be displayed in the Help Documentation Search Results, as a brief synapse of the document that users can read before actually opening the document.

Example:


<page>
    <topic>Adding a Description Section</topic>
    <description>Adds a description section to a page 
    using the <![CDATA[<description>]]> tag.</description>
</page>

To add multiple paragraphs to a description, use the <p></p> tag:


<description>
    <p>Paragraph 1</p>
    <p>Paragraph 2</p>
</description>

Adding a Discussion Section

A discussion section is a lot like the description section only that it is longer and doesn't appear in the Help Documentation Search Results when the document is queried. This is for cases where you need a paragraph or so to describe something but do not have any images or code examples to add to the text. Use the <discussion></discussion> tag pair to create the section. The title 'Discussion' is created automatically once the code is parsed. Alternately, you can create a similar section using the <section></section> tags and giving this the title 'Discussion' and then a <description> and so forth.

Example of Simple Discussion:


<page>
    <topic>BLOB.DUMP</topic>
     <description>Dump the contexts of a blob to HEX 
    (optional format hex per line, for hex/ascii etc, numbering).</description>
    <discussion>
    The .DUMP() method dumps the contents of a blob using the specification provided 
    by the Format_String. For example, the following Xbasic code opens a temporary 
    Alpha Anywhere query file and dumps out its contents.
    </discussion>
</page>

Alternatively you could do the same thing without <discussion></discussion> tags:

Example:


<page>
    <topic>BLOB.DUMP</topic>
     <description>Dump the contexts of a blob to HEX 
    (optional format hex per line, for hex/ascii etc, numbering).</description>
    <sections>
        <section>
            <title>Discussion</title>
            <description>
            The .DUMP() method dumps the contents of a blob using the specification provided 
            by the Format_String. For example, the following Xbasic code opens a temporary 
            Alpha Anywhere query file and dumps out its contents.
            </description>
        </section>
    </section>
</page>

To add multiple paragraphs to a discussion, use the <p></p> tag:

Example:


<discussion>
    <p>Paragraph 1</p>
    <p>Paragraph 2</p>
</discussion>

The Syntax Section:

Adding functions, objects, and namespaces to the help system often requires the description of their syntax. In this case use the <syntax></syntax> tags. The tags are placed directly below the <topic></topic> tags on a page. The syntax sections title will be added automatically once the page is parsed.

Example:


<page>
    <topic>RTF_TO_TEXT Function</topic>
    <syntax>Text_String as C = rtf_to_text(B field_data )</syntax>
</page>

Multiple Syntax Definitions:

Sometimes a function contains more than one syntax definition. In this instance the <prototypes></prototypes> tags are used to contain the multiple definitions while the <prototype></prototype> tags are used to actually describe the definitions themselves. As with the syntax tag the <prototypes> tags go directly after the <topic tags> and will automatically get a title after the code is parsed. The result looks something like this...

Example:


<page>
    <topic>CDOW Function</topic>
    <prototypes>
        <prototype>Day_of_Week as C = CDOW(D date[,L localized])</prototype>
        <prototype>Day_of_Week as C = CDOW( D time as T [, localized as L])</prototype>
    </prototypes>
</page>

Adding an Arguments Section:

The <arguments></arguments> tags are used to add an arguments section just below the syntax tags, to describe any arguments contained in the syntax. Individual arguments are assigned using the <argument></argument> tags. Each Argument tag pair can the contain <name></name> tag pair describing the name of that argument and a <description></description> tag pair that describes the meaning of the argument. You can add as many <argument></argument> tags inside the <arguments></arguments> tags as are required.

Example:


</page>
    <topic>A5_TIMESTAMP Function</topic>
    <syntax>Formatted_Short_Time as C = a5_timestamp()</syntax>
    <arguments>
        <argument>
            <name>Formatted_Short_Time</name>
            <description>The current system time formatted according to your 
            preferences.</description>
        </argument>
    </arguments>
</page>

Adding a Properties Section:

The <properties></properties> tag pair allows you to list properties. Individual properties are defined using the <property></property> tag pair. Each property can contain <name></name>, <type></type>, <description></description> tag pairs.

Example:


<page>
    <properties>
        <property>
            <name>GridCache</name>
            <type>Alpha5DotNet.DataCache</type>
            <description>Read Only</description>
        </property>
    </properties>
</page>

You can add a generated="true" to a property.

Example:


<page>
    <properties>
        <property generated="true">
            <name>ActiveSheetIndex</name>
            <type>N</type>
            <description>Numeric ActiveSheetIndex property</description>
        </property>
    </properties>
</page>

Adding a Methods:

You can add a Methods section to your document if you are adding a

Example:




<methods>
    <method>
        <syntax></syntax>
        <description></description>
        <arguments>
            <argument>
                <name>Context has no methods.</name>
                <description></description>
            </argument>
        </arguments>
    </method>
</methods>

You can add a generated="true" to a method.

Example:


<page>
    <methods>
        <method generated="true">
            <name>AddFont</name>
            <syntax>.AddFont as Office::Font ([InitialFont as P])</syntax>

            You can pass an existing font to copy its values into the new one.</description>
        </method>
    </methods>
</page>

Links, Formatting Directives, and Special Characters

Adding In Text Links:

The <links> tags are added near the end of the page (just before the <page> tag and after </sections> or </groups>). The <links></links> tag pair contains a list of all of the other pages or images that the document links to. Each individual link is wrapped in it's own <link></link> tag pair like so...

Example:



<links>
    <link href="./Installation/index.xml">Alpha Anywhere Application Server for IIS Installation Guide"></link>
</links>

Once the link is defined you can add square brackets [ ] around any text inside a <description></description>  tag pair to create a link to the defined page. The text inside the square brackets must match exactly the text between the <link></link> tag pair in order for the link to work.

Example:


<description>[Alpha Anywhere Application Server for IIS Installation Guide]</description>

Adding In Image Links:

You can add images inline by adding an href to the <link></link> tag pair and defining a link to an external image.

Example:


<links>
    <link href="images/Close_button.gif">x_img</link>
</links>

Once the link is created, you can add the image to text by using square brackets [ ] around your defined tag, in this case 'x_img'. This should be added inside a <description></description> tag pair.

Example:


<description>Click the close button [x_img] and save.</description>

Escaping Special Characters

Some characters, such as angle brackets and ampersands, will cause the xml parser to fail. To get around this limitation, use the <![CDATA[ ]]> tag. The <![CDATA[ ]]> can be used inside any tag. If possible, however, it is best to avoid using <![CDATA[ ]]> in the <topic>, <shortlink>, and the top-level <description> tag that contains the page's description.

If you use the <![CDATA[ ]]> inside a link (see Adding In Text Links), you must include the <![CDATA[ ]]> as part of the link definition.

Example:


<page>
    <topic>Using the Documentation XML</topic>
    <sections>
        <section>
            <description>Check out the [<![CDATA[<TBL>.OPEN Method]]>].</description>
        </section>
    </sections>
    <links>
        <link href="/documentation/index?search=table open function"><![CDATA[<TBL>.OPEN Method]]></link>
    </links> 
</page>

Adding Text Emphasis

The *[directive:textToEmphasize]* syntax can be used to add emphasis to text. Empasis can simply be to callout the text as bold. It can also be used to call out text as Javascript or Xbasic code as well as identify text as a UI element.

The directive is optional. It is not required unless textToEmphasize contains a colon character. If you do not include a directive, omit the colon completely.

Below is a list of options available.

*[this is my text]*
Displays "this is my text" using the emphasis-bold class.
*[bold:this is my text]*
*[bold: ]* is the same thing as *[ ]*. Displays "This is my text" using the emphasis-bold class. Use *[bold:]* in the event that your text contains a colon.
*[xb:ui_msg_box(msg)]*
Used to indicate the text is xbasic code. Displays "ui_msg_box(msg)" using the emphasis-xb class.
*[js:{dialog.object}.submit();]*
Used to indicate the text is javascript code, eg: JSON, javascript or node.js code. Displays "{dialog.object}.submit();" using the emphasis-js class.
*[ui:Display format]*
Used to indicate the text refers to a User Interface element, such as a property, menu option, name of a popup dialog, or button. Displays "Display format" using the emphasis-ui class.

New emphasis options can be added if needed if the existing emphasis options are inadequate for expressing what needs to be emphasised - for example, if we wanted to callout node.js code separately from client-side javascript, then it may be appropriate to add a "node" directive, eg *[node:this is node.js code]*.

The visual effectcs are defined in the helpserver/assets directory inside the theme.css file. If you add a new empasis directive, you must also add a new CSS class defining the desired style for the empasised text.

If you add a new emphasis option, remember to also update this document.

Adding Inline and External Links

In addition to the emphasis classes listed above, the *[ ]* syntax can also be used to add links. There are two types of links available - regular links and external links. Both links have the format *[linkDirective:TextToShow@PageURL]*.

Both the linkDirective and PageURL are required. TextToShow is optional. Use the "@" symbol to indicate where the TextToShow ends and the PageURL begins. If your link does not include TextToShow, do not include the "@" symbol in the link definition.

Below are the available linkDirectives:

*[link:TextToShow@PageURL]*
Defines a link. The link is opened in the current window or tab.
Examples:
*[link:Alpha Software@http://www.alphasoftware.com]*
*[link:http://www.alphasoftware.com]*
*[extlink:TextToShow@PageURL]*
Defines an external link. An external link is opened in a new window or tab.
Examples:
*[extlink:Google Search Engine@http://www.google.com]*
*[extlink:http://www.google.com]*

Searching: Search Parameters

There are several search parameters availalbe. These must be typed into the address bar for the search page:

display=titles-only
Shows search results, but only lists page titles
search=titles
Searches only titles - ignores page content
limit=N
Number of results to show, where "N" is a numeric value greater than 1. EG, limit=10000

Page Behaviors

Flatten

Some API pages include a button to "flatten" the page. The "flatten" function grabs all the documentation in the child directories and produces a single page. See Context Object for an example.

Flatten is explicitly defined in the helpserver.js and helpnosearch.js files. To add flattening for your tree, add the file path to your document to the canFlatten function. EG:


    var flattenChildren = [
        { path : "/Ref/Api/Functions/" , level : 2 },
        { path : "/Ref/Api/Namespace/" , level : 1 },
        { path : "/Ref/Api/Objects/" , level : 1 },
        { path : "/Ref/Api/My/New/API", level : 1}
    ];

Unflatten

Unflatten is the complement to flatten. It is automatically included as an available feature if a page can be flattened.

Print

Any page in the documentation system can be printed. The "print" feature opens the page in a new window and automatically opens the Print dialog. Print is available for flattened pages, as well, allowing you to print a flattened API for reference.

Share

All pages can be shared. The share feature uses a "shortlink", which is defined by the page's topic or a special shortlink defined with the <shortlink> XML tag. Shortlinks are defined in the links.json file. If you add a file that doesn't have a shortlink, you can manually update the links.json file.

Share links must be unique. If another page is already listed in links.json that uses your page's topic, use the <shortlink> XML tag to add a unique shortlink for your page.

It is important that the <shortlink> tag is used as the links.json file is periodically automatically updated. If your shortlink is not a page topic and is not defined in a <shortlink> tag, you risk having your page's shortlink removed from the system.

Moving Files

IMPORTANT: If you move files, make sure you update their shortlink in links.json. Some documentation files are linked directly into Alpha Anywhere using the shortlink. If you move a file and do not update the shortlink, you will break Alpha Anywhere.
