Structuring Sets and Normalizing Tables

Before creating a set, you should first design (or restructure) your tables so that the set you design accurately models your data. The process of determining this structure is called normalization. While this may sound technical, there are really only a few things you need to learn to successfully normalize the data of your set. Some of the rewards of a well-structured set are greater speed, more flexibility in querying, and greater reporting capabilities.

This section teaches you the rules of normalization by providing you with a common example, an invoice.

The Invoice Example

Suppose you want to keep track of invoices. For each invoice, you want to know the name of the customer for whom the invoice was created, you want to know which items were purchased and how many, and you want information about the invoice itself, such as the date on which it was entered and by whom. Note that the Invoice table does not contain information about the description, prices, or quantities of the items purchased. The Invoice table also excludes all the customer fields, except the customer number. To normalize the data, the excluded fields are placed in separate tables.

By putting customer information in a separate table, you can eliminate the storage of redundant information. If the same customer has more than one invoice, you do not need to enter the customer information multiple times; all you to need to do is enter the customer number and a match is found in the customer table.

Suppose you discover that a customer's name was spelled incorrectly, you can make the change to the customer record, and the invoices associated with the customer are automatically changed as well. Had you put the customer-information in the invoice table, you would need to change each of the customer's invoices, which is both more time consuming and more prone to error.

Information about what the customer purchased is placed in a separate table for a different reason. Because invoices can contain different numbers of line items, if you placed them all in the same table, you would need to create enough fields to handle the maximum possible invoice item entries. For example, you decided that you would never put more than 25 different items on an invoice, you would need to create fields to handle this:
