Known Issues

This page lists the currently known issues in Alpha Anywhere that are open. Issues are tracked using GitHub. You will find a list of "labels" for each issue. These lables indicate the associated feature in Alpha Anywhere and the type of issue - bug, enhancement, etc. Clicking a label will display a list of all open issues with the same label.

For a list of all issues, including issues that are closed, visit the Alpha Anywhere GitHub Issues List.

#14 Publish Error: Cannot connect to the remote computer. Make sure Web Deploy is installed

Labels: Application Server for IIS, bug

After following the Alpha Anywhere Application Server for IIS Installation Guide publish to IIS fails with an error saying to check that Web Deploy is installed and that the Web Management Service is started:

Sometimes Web Deploy does not get fully installed or, as the error message says the "Web Management Service" is not started.

The installer should be modified to ensure Web Deploy is fully installed and the needed services are running.

Workaround: Ensure that the Windows services "Web Management Service" and "Web Deployment Agent Service" are running.

If the Web Management Service is not started, it can be enabled and started from IIS Manager. Open the Management Service feature,

and then enable and start the Management Service.

If "Web Deployment Agent Service" is missing, go to the Windows Control Panel - Programs and Features, find Microsoft Web Deploy 3.5, right click and then select change. Go to the next wizard page and you'll then need to click a Change button. Make sure all the features are installed. It should look like this:

If any of the items have a red X change them so that they are "Will be installed on local hard drive".

#15 Publish to a site's root application will remove any sub applications in the site.

Labels: Application Server for IIS, bug

If you have a site with a sub application as well as a root application, publishing to the root application will cause all sub applications to be removed. A root application of a web site is what will be served when handling a request with no path e.g. "http://www.mycompany.com/". A web site can also have sub applications. A sub application name is included in the request path e.g. "http://www.mycompany.com/app1". Alpha Anywhere Developer Edition will publish to the root application when the Application name in the publish profile is blank. When an application name is included in the publish profile ("app1" in url above), that name is used as the application name in IIS.

Workaround: Republish the sub applications after publishing the root application.

#18 Some file names cause errors with web.config authorization rules

Labels: Application Server for IIS, bug

Publish to IIS will stage all files in a workspace's folder include files added by source control systems like CVS. Deny rules are generated for these files in the web.config file. Some of these files have special characters in them and while the file names are legal, the special characters are not allowed in the authorization rules (deny rules). One example is with CVS that creates a file like this: ".#index.a5w.1.1". The leading period is not allowed in an authorization rule. There needs to be a way to exclude files from being included in the publish.

#29 Add setting to set characters that are not allowed in a query string.

Labels: Application Server for IIS, Developer Edition, enhancement

The list of characters that are not allowed in a query string under IIS is hard coded into the web.config file in the requestPathInvalidCharacters attribute of the httpRuntime element. This is a comma separated list and is set to "<,>,*,%,&,\,?" without the double quotes (and is HTML encoded in the web.config file). Legacy applications may need to allow some of these characters.
