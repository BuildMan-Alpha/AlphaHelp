Calling a UDF from a Report Printed by an .A5W Page

This information applies if you have an Alpha Anywhere desktop application in which:

  • you have defined one or more User Defined Functions (UDFs)

  • you have created a report and the report has calculated fields or code in an event handler that uses one of the UDFs defined in the database

If you build a web application that prints this report to file as a PDF file, you need to ensure that the UDFs are available when the report prints. When a web application is running, there is no guarantee that the database (i.e. the .ADB file) in which the UDFs were defined will be open.

If you know that the .ADB file in which the UDFs were defined is in fact going to be open, then this document does not apply to you, because the UDFs will automatically be available.

The Solution

  1. When you publish your application, publish the library files where the UDFs are stored (typically these will be <databasename> with the  .ALB, .ALM and .ALX file extensions). You can publish these files to the same folder where you store your data files, or to the folder in the webroot where application is published.

  2. If you publish these library files to the folder containing your .a5w pages, put this code into the report's OnPrintInit event.

    udf_library_load("<databasename>.alb",global_variables())

    where <databasename> is the name of the database.

A shared server is a server at an ISP. On a shared server you do not have control over stopping and starting the server. If you are not using a shared server, an alternative solution is to create a compiled .AEX file with your UDFs. You would place the .AEX file in the addins_installed folder under the folder where the server is installed. You will need to exit and restart Alpha Anywhere in order to load the .AEX file. See Compiling Scripts and Functions for more information.
