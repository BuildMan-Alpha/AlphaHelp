Using Variables on A5W Pages

An A5W page is really a script that contains HTML, Xbasic (and optionally JavaScript). This script runs on the Application Server, which generates HTML and sends it (and any option JavaScript) to the user's browser.

Variables Available When the Script Loads

By default there are three sets of variables available to an A5W script.

  • Each input control creates a variable. The name of the variable is set by the &ltINPUT> control's NAME argument. Hidden input controls can define variables that the user cannot see or change. These variables exist until the user sends a response to the Application Server and/or loads a different script.

  • The Application Server creates a Session object when a client (browser) establish a session, within which you may create session variables. A session variable exists until the session expires. See also Using a Session Variable in a Component.

  • The Request Object is a server object that contains information about the URL that the user has sent to the Application Server. Your Xbasic scripts can refer to the contents of the Request Object.

  • The Result Variable is available for scripts to cancel the CanUpdateRecord, CanDeleteRecord, and CanInsertRecord grid events.

Variables that You can Create

In addition, you can create additional variables with Xbasic code embedded in a script.

  • Any variable that you create with a DIM statement will be available within the script or function only and will disappear when the user loads a different script.

  • Any variable that you create with a DIM statement and prefix with " session. " will be available within all scripts for the duration of the user's session.

  • Protected variables are a type of session variable that cannot be set with a URL, but only with Xbasic.

  • Cookies are files that save the values of variables between sessions. Because some users prevent your browser from saving cookies, or delete cookies between sessions, cookies are not a reliable way to preserve information.

Passing Variables to a Another Page

There are three ways to pass (non-session) variables from one script to another.

  • The RESPONSE.REDIRECT() method can pass one or more variables as arguments appended to the target URL.

  • A hyperlink can pass one or more variables as arguments appended to the target URL. All variables created as part of a URL are of type character.

  • The &ltFORM> object passes a variable defined by the NAME argument to the script defined by the ACTION argument.

Modifying Grid Properties

There are many other variables (properties) associated with grid components. You may modify these variables when you load a script. Each grid component has a section of its Xbasic code that looks something like this.

...

<%a5

Delete Tmpl

DIM Tmpl as P

tmpl = a5w_load_component(" ComponentName ")

...

with tmpl

    componentName = " ComponentName "

end with

...

%>

...

Inside the with tmpl ... end with structure you may set the value(s) of grid properties. For example, the following code redefines the value of the tmpl.dbf.filter property. You can see the set of grid properties on the Xbasic tab of the Grid Component Builder.

...

with tmpl

    componentName = " ComponentName "

    if eval_valid("cust_id")

        dbf.filter = "customer_id = '" + cust_id + "'"

    end if

end with

...

Reading and Referencing Grid Values

A grid contains one or more < INPUT > controls. Typically, a grid contains an array of < INPUT > controls, many controls on a line, and many lines in the grid. When a script runs, you can reference the values of the controls on the current line. You set the names of the variables in the Grid > Fields page of the Grid Builder. You can also read the names of the variables by looking for the .varname properties on the Xbasic tab.

In Alpha Anywhere you can read the values of the other (non-current) rows of the grid. There are 3 sets of variables that can be read or set when grid events occur. These are:

  • Args - variables associated with all fields on all rows of the grid

  • DataSubmitted - variables associated with the current record

  • PageVariables - variables that contain various page and grid properties

  • SearchDataSubmitted - variables populated by the search part of a grid

You have the opportunity (depending on the event) to read or change grid values before the page is submitted to the Application Server.
